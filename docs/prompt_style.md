# Prompt Style Guide для Wave-промптов

Эталонный формат промпта, который выдаётся Claude'у для исполнения
целого этапа работ (Wave / sub-wave). Применяется ко всем Wave'ам
NETRUN-проекта: B-*, Wave C, Wave D.

Цель: один промпт = одна логическая Wave. Claude работает автономно
от начала до коммитов, СТОП перед push, ждёт явного «ок пуш».

## 1. Общая структура

```
[Header]                ─── 1 параграф: Wave-id + цель в 2-3 строках
[Правила автономности]  ─── boilerplate (см. § 3)
[ЭТАП 1] … [ЭТАП N-3]   ─── собственно работа
[ЭТАП N-2] Verify       ─── 4 команды, ожидаемые counters
[ЭТАП N-1] Auto-commit  ─── логические коммиты, по одному на группу
[ЭТАП N]   Финальный отчёт + СТОП перед push
```

Этапы разделяются строкой из `═` (длина ~71). Это визуальный маркер
boundaries.

## 2. Header

Один абзац, без подразделов. Содержит:
- Wave-id и название (`Wave B-7a — production hardening: ...`)
- Цель в 2-3 строках, бизнес-формулировкой (что станет возможным)

## 3. Правила автономности (boilerplate)

Стандартный блок, копируется между Wave'ами:

- Макро-режим: ЭТАП 1 → ЭТАП N без round-trip
- Auto-commit per logical group
- Push gate: НИКОГДА не push автоматически
- Pin gate: requirements.txt / pyproject.toml не трогать без явного
  разрешения в этапе
- Behavior gate: pytest зелёный после каждого этапа
- Scope gate: только то что в Wave, никаких побочных features
- Lint-в-отдельный-коммит: побочные lint-фиксы → свой коммит, не
  бандлить в docs или фичевый

## 4. Этап (anatomy)

- Один этап = одна логическая единица
- Полные пути от корня репо
- SQL inline, полностью
- Если файл создаётся с нуля — давать полное содержимое или скелет
- Если изменяется — указать точку вставки

## 5. Verify-блок

Стандарт:

```
.venv/Scripts/python.exe -m ruff format --check .
.venv/Scripts/python.exe -m ruff check .
.venv/Scripts/python.exe -m mypy orchestrator/ shared/
.venv/Scripts/python.exe -m pytest tests/ -q
```

Указывать ожидаемые counter'ы (было N → стало N+K). Без них Claude
может не заметить недостающий файл.

## 6. Auto-commit

Правила:
- Один коммит = один логический слой
- Префикс по area: `watchdog:`, `deploy:`, `scripts:`, `api:`,
  `docs:`, `tests:`, `migrations:`, `metrics:`, `admin:`, `hardening:`,
  `schemas:`, `lint:`, `deps:`, `design:`, `logging:`
- Глагол повелительный: `add`, `switch`, `remove`, `fix`, `close`
- ≤ 80 символов в subject
- НИКОГДА `--amend` после первого push
- НИКОГДА `--no-verify` без явного разрешения

## 7. Финальный отчёт + СТОП

Доложить:
- Все SHA новых коммитов (таблицей с subject'ами)
- Created files (с путями от корня)
- Modified files
- Verify-статусы (4 строки)
- Wave/sub-wave CLOSED
- "Ahead of origin/main by M commits"
- Жду «ок пуш»

НЕ ДЕЛАТЬ git push. Точка.

## 8. Push gate, Pin gate, прочие гейты

| Гейт | Что не делать без явного «ок» |
|---|---|
| Push gate | `git push` любого вида |
| Pin gate | менять requirements.txt / pyproject.toml |
| Migration gate | запускать migrate.py против production |
| Destructive gate | `git reset --hard`, `rm -rf` непомеченных директорий |
| Force-push gate | `git push --force` на любой бранч |

Если этап ТРЕБУЕТ перешагнуть гейт — это явно прописывается:
«ЭТАП X — pin gate ослаблен, разрешено добавить X в requirements.txt».

## 9. Single-block format

Промпт пишется как ОДИН triple-backtick блок. Без markdown-обёртки
сверху, без замечаний внутри. Пользователь копирует кнопкой
одним кликом.

Любые комментарии/caveats/уточнения — В отдельном сообщении ДО или
ПОСЛЕ блока, никогда внутри.

## 10. Gotchas

- Truncation: prompt > 500 строк — рискует обрезаться. Сжимать
  boilerplate, выносить большие SQL/file-content в Appendix.
- Russian narrative + English code/SQL/paths.
- Поведенческие assumptions писать явно (например: «SQL опирается на
  то, что worker.py обновляет updated_at при claim'е»).
- Test counts: указывать «было N → стало N+K». Без указания Claude
  может «починить» падающие тесты их удалением.
- Идемпотентность install-скриптов: `bash install.sh && bash install.sh`
  должен дать exit 0 на втором запуске.
- /opt/<project> на production-нодах НЕ git-repo (install копирует
  без .git). `git pull` там не работает.

## 11. Skeleton-шаблон

Минимальный каркас для нового Wave-промпта:

```
Wave X-Y — <название>.

Цель: <2-3 строки>.

═══════════════════════════════════════════════════════════════════════
ПРАВИЛА АВТОНОМНОСТИ
═══════════════════════════════════════════════════════════════════════

[см. § 3]

═══════════════════════════════════════════════════════════════════════
ЭТАП 1 — <название>
═══════════════════════════════════════════════════════════════════════

[действия]

[...]

═══════════════════════════════════════════════════════════════════════
ЭТАП N-2 — Verify
═══════════════════════════════════════════════════════════════════════

[см. § 5]

═══════════════════════════════════════════════════════════════════════
ЭТАП N-1 — Auto-commit (M логических коммитов)
═══════════════════════════════════════════════════════════════════════

[см. § 6]

═══════════════════════════════════════════════════════════════════════
ЭТАП N — Финальный отчёт + СТОП перед push
═══════════════════════════════════════════════════════════════════════

[см. § 7]
```

## 12. Эталоны

- Wave B-7b.1 — образец sub-wave с per-file rename коммитами
- Wave B-7b.2 — образец metrics/instrumentation Wave
- Wave B-7b.3 — образец features-with-Pydantic Wave

Если новая Wave заметно отклоняется от формата (например, кросс-репо
в Wave C) — обновлять этот гайд первым, потом писать промпт.
